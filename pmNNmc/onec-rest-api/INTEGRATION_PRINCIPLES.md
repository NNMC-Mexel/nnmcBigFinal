# Принципы интеграции NNMC с 1С

Этот файл фиксирует договоренности по интеграциям корпоративной системы с 1С.
Главное правило: сначала подтверждаем точные объекты и реквизиты в 1С, потом
пишем endpoint.

## Общий подход

| Этап | Что делаем | Зачем |
| --- | --- | --- |
| 1 | В 1С создаем расширение конфигурации | Чтобы не кастомизировать основную конфигурацию |
| 2 | В расширении создаем HTTP-сервис `NNMC` | Это REST API внутри 1С |
| 3 | В HTTP-сервис добавляем URL-шаблоны и методы | Каждый метод отвечает за одну бизнес-задачу |
| 4 | Публикуем расширение на IIS | Чтобы корпоративная система могла обращаться к 1С по HTTP |
| 5 | Корпоративная система отправляет запрос | Например: получить сотрудников, табель или создать документ |
| 6 | 1С внутри метода работает со справочниками, документами и регистрами | 1С сама ищет и проверяет нужные данные |
| 7 | 1С возвращает JSON или создает документ | Портал получает результат и показывает пользователю |

## Что нельзя забывать

- Работаем через расширение 1С, не меняем основную конфигурацию.
- HTTP-сервис называется `NNMC`.
- Корневой опубликованный URL примерно такой:

```text
/copy10062025-rest/hs/nnmc
```

- Логика всегда такая:

```text
Портал -> REST API 1С -> данные внутри 1С -> JSON / документ 1С
```

- Для чтения данных используем запросы 1С:

```bsl
Запрос = Новый Запрос;
Запрос.Текст = "...";
Результат = Запрос.Выполнить();
```

- Для создания документов используем:

```bsl
ДокументОбъект = Документы.ИмяДокумента.СоздатьДокумент();
```

- Никогда не угадываем названия реквизитов. Проверяем в конфигураторе:

```text
Документ / Справочник / Регистр -> Данные -> Реквизиты / Табличные части
```

- Если 1С пишет ошибку при ручном сохранении документа, это важная информация:
  по ней определяем обязательные поля.

## Уже используемые методы

| Метод | Назначение |
| --- | --- |
| `GET /v1/employees` | Сотрудники из 1С |
| `GET /v1/timesheets` | Список табелей |
| `GET /v1/timesheet` | Один табель |
| `POST /v1/kpi-accruals` | Создание разового начисления KPI |
| `GET /v1/employee-cards` | Карточки сотрудников |
| `POST /v1/vacation-requests` | Создание черновика заявки на отпуск |

## Уже подтвержденные объекты 1С

### Сотрудники

```text
Справочник.ФизическиеЛица.ИИН
Справочник.Сотрудники.ФизическоеЛицо
РегистрСведений.ТекущиеКадровыеДанныеСотрудников
```

Актуальные кадровые данные лучше брать из регистра:

```text
РегистрСведений.ТекущиеКадровыеДанныеСотрудников
```

Важные поля:

```text
ФизическоеЛицо
Сотрудник
ГоловнаяОрганизация
ТекущаяОрганизация
ТекущееПодразделение
ТекущаяДолжность
ДатаУвольнения
ДатаПриема
ТекущийВидЗанятости
КоличествоСтавок
```

Важный нюанс:

```text
Одно физическое лицо = один ИИН.
У одного ИИН может быть несколько сотрудников / табельных номеров.
Основное место работы определяем по виду занятости: "Основное место работы".
```

### Табель

```text
Документ.ТабельУчетаРабочегоВремени
Табличная часть: ДанныеОВремени
```

### KPI

```text
Документ.РазовоеНачисление
Вид начисления: KPI
```

### Отпуска

Для нового endpoint по отпускам логика:

```text
Портал отправляет заявку на отпуск.
1С ищет физлицо по ИИН.
1С находит активного сотрудника.
1С находит вид отпуска.
1С создает черновик Документ.ОтпускаСотрудников.
Портал получает номер, ссылку и статус.
```

Используемый документ:

```text
Документ.ОтпускаСотрудников
```

Виды отпусков:

```text
Справочник.ВидыОтпусков
```

Поле типа отпуска:

```text
СправочникСсылка.ВидыОтпусков
```

Если доступен `АУ_ИдентификаторSimbase`, можно сопоставлять типы отпусков по
нему. Если он пустой, временно сопоставляем по наименованию.

## BPM backend

BPM вынесен в отдельный Strapi backend:

```text
server-bpm
```

Требования:

- `server-bpm` работает на PostgreSQL, не на SQLite.
- Вход через единый Keycloak.
- Для нового BPM backend используется отдельный токен `bpm_token`.

Frontend env:

```env
VITE_BPM_API_URL=http://192.168.101.25:12016
```

1C REST env:

```env
ONEC_API_URL=http://192.168.40.83/copy10062025-rest/hs/nnmc
ONEC_API_USER=Администратор
ONEC_API_PASSWORD=...
```

Для отпусков можно переопределить полный URL:

```env
ONEC_VACATION_REQUEST_URL=http://192.168.40.83/copy10062025-rest/hs/nnmc/v1/vacation-requests
```

## Перед новым endpoint

Перед написанием нового endpoint в 1С:

1. Подтвердить объект: документ, справочник или регистр.
2. Подтвердить точные реквизиты.
3. Подтвердить табличные части.
4. По возможности вручную создать документ и нажать `Записать`.
5. Если 1С ругается, сохранить полный текст ошибки.
6. Только после этого писать обработчик HTTP-сервиса.
